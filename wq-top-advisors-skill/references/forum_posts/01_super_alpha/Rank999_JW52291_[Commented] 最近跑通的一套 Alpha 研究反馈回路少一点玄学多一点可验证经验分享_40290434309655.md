## 帖子正文

最近读到 OpenAI 的一篇文章《Harness engineering: leveraging Codex in an agent-first world》，里面有一句话很适合迁移到 Alpha 研究里：人负责设定方向和约束，agent/工具负责执行和验证。

我把它翻译成 WorldQuant BRAIN 的语境，大概是：

> 不要只让工具“多跑一些表达式”，而是给它一个可检查、可复盘、会自我纠错的研究回路。

这篇帖子不是分享一个可以直接复制的 Alpha，而是分享我最近跑通的一套流程。它对我最有帮助的地方是：当一个方向失败时，我能更快判断“是字段没信号”，还是“表达式结构错了”，还是“本地通过但 prod correlation 太拥挤”。

1. 先把研究任务写成 contract，而不是一句模糊 prompt

以前我经常这样开始：

“帮我找一个 MEA/TOP300/D1 的 Model alpha。”

后来我改成更像 contract：

- Region / Universe / Delay / Neutralization 明确写死

- 数据集或字段范围明确

- 硬门槛明确，例如 Sharpe、Fitness、Margin、Self Corr、Prod Corr

- 每轮最多并发多少条，避免一次性生成太多相似壳

- 哪些 warning 可以接受，哪些必须先修

- 什么情况下继续微调，什么情况下必须换 dominant field / structure

这样做的好处是：工具不再只是在“生成更多表达式”，而是在执行一个可以审计的实验协议。

2. 每轮只问一个结构性问题

我现在更喜欢小批量、多轮、强记录。

例如最近一条 MEA / TOP300 / D1 / Model 路线，我不是一次扔几十个相近表达式，而是每轮问一个明确问题：

- `star_eq_rank` 是否有原始信号？

- 简单 time smoothing 能不能过 IS？

- 如果 self 过了但 prod-corr 高，是阈值问题还是 crowded shell？

- 换成同类组件字段，例如 cashflow / OPE / accruals，会不会改善？

- MARKET / SECTOR / NONE neutralization 是谁在伤害信号？

- `trade_when` 稀疏化是在修 turnover，还是只是制造另一个高相关壳？

每一轮结束后我只记录四种状态：

- 绿灯：可以进入下一道 gate

- 黄灯：接近门槛，但必须说明修哪个 blocker

- 红灯：同结构不要再微调

- 基础设施问题：超时、请求排队、平台 backpressure，不计为 Alpha 失败

这个分类非常重要。因为很多时候我们输在“把基础设施问题当成信号问题”，或者“把 prod-corr 红灯当成窗口没调好”。

3. 最近一次实战：不是一直调窗口，而是承认 prod-corr 红灯

这次路线里，早期有几个候选看起来很诱人。

一个 `star_eq_rank` 相关的候选，IS 指标不错，self 也能过，但 prod correlation 到了大约 `0.94`。后来另一个 near-gate 候选通过了 self，修过 2Y / turnover 之后也能进入 prod 检查，但 prod correlation 还是大约 `0.91`。

如果只看 IS 或 self，会很容易继续调：

- 换窗口 2/3/4/5

- 改 rank / quantile

- 调 threshold

- 加一点 decay

- 再试一个相邻字段

但我的结论是：这不是“还差一点”的问题，而是同一类拥挤结构的问题。

所以我把这一支标成：

> Quality/self 绿灯，prod-corr 红灯。不要继续做同壳微调，必须换结构或换 dominant leg。

这个判断节省了很多时间。

4. 真正突破来自 setting + structure 的组合变化

后面我转向了 cashflow 质量相关字段。它在 MARKET neutralization 下只是黄灯，Sharpe 不够高，但 Fitness、2Y、Margin 都不错。

关键变化不是继续调窗口，而是做了两件事：

1. 把 neutralization 从 MARKET 改成 NONE，观察是否是中性化过度压掉了信号；

2. 用 `trade_when` 做稀疏化，只保留更强的 cashflow quality 状态。

最后出来的实际结果：

- Sharpe 约 `2.21`

- Fitness 约 `2.37`

- Margin 约 `0.014`

- Self correlation 约 `0.26`

- Production correlation 约 `0.64`

这才进入了 pre-submit 候选池。

我故意没有贴完整字段名和 alpha id，因为我觉得真正可复用的不是这个表达式，而是这条路径：

> 当同一壳的 prod-corr 连续红灯时，不要继续微调阈值；换 dominant field、换 setting、换结构，并用下一轮实验验证这个判断。

## 5. 我现在会强制记录三类“失败”

我以前的问题是只记录成功候选，导致下次又重复踩坑。现在失败也要入账，而且要分类型。

第一类：质量失败。

例如 Sharpe / Fitness / SubU / 2Y 明确不行。这种通常说明表达式经济含义或稳定性不足。

第二类：相关性失败。

例如 self 或 prod correlation 太高。这时不要只说“失败”，而要记录它到底是 self 高，还是 prod 高。self 高通常说明撞到自己已有 alpha；prod 高更像撞到公共拥挤结构。

第三类：门槛边缘失败。

例如 prod correlation 在 `0.70-0.73` 附近，我不会马上放弃，而是当成 near-gate rescue；但如果同结构直接到 `0.85+` 或 `0.90+`，我一般会标红，不再做同壳微调。

这类记录比单纯保存表达式更值钱。

6. 另一个教训：不要相信“看起来通过”，要相信最终数字

我还有一次反例：一个 USA Sentiment 路线曾经被误记录成 pre-submit，因为工具流程里先看到了 prod request 成功和 gate 标记。但后来复核实际 `prod_corr_max` 大约是 `0.765`，超过了 `0.70` 的门槛，所以必须撤回，把状态改成 failed。

这个错误让我加了一条规则：

> prod request 成功不是 prod-corr 通过。最终只相信明确的数值门槛。

这点对半自动化研究尤其重要。工具可以提高吞吐，但 gate 必须机械、明确、可复核。

7. 我的当前 Alpha Harness 模板

如果你也想把自己的研究流程变得更稳定，可以试试这个模板：

1. 写 route contract

region / universe / delay / dataset / constraints / gates

2. 每轮设计 2-3 个结构问题

不要只做同一表达式的参数扫描

3. 小批量 multisim

每轮结束后只保留代表候选

4. 先 IS，再 self，再 prod

prod 调用很贵，不要给每个黄灯候选都跑

5. 每轮写 ledger

winner / nearline / closed branch / blocker / next question

6. 红黄绿灯决策

质量红、self 红、prod 红、基础设施问题要分开

7. 定期清理路线记忆

过时结论要撤回，错误入库要修正

我感觉这比“多跑一点”更重要。因为 Alpha 研究最大的问题往往不是计算不够，而是注意力被太多相似失败分散了。

8. 最后：把 AI 当执行器，不要把判断外包

OpenAI 那篇文章讲的是软件工程，但我读完后最大的启发是：真正的杠杆不是让 agent 替你“想一个答案”，而是给它一个能反复执行、能暴露错误、能把经验沉淀下来的系统。

在 Alpha 研究里也是一样：

- 人来决定经济假设

- 工具负责批量验证

- 日志负责保留事实

- gate 负责防止自欺

- 记忆负责避免重复失败

如果这个流程搭起来，哪怕某一轮没有找到 Alpha，也不是完全浪费。因为你至少知道了一个方向为什么不行，以及下一轮应该换什么。

我现在越来越相信：好的 Alpha 不是单次灵感的产物，而是一个反馈回路长期压出来的结果。

欢迎大家交流：你们在 BRAIN 上最容易重复踩的坑，是 quality 不够、self correlation、prod correlation，还是 LOW_SUB / turnover 这类边缘门槛？
