### 二、实战发现：框架的不足之处

#### 问题1：信号灯判断过于保守

**现象** : 第二轮最高Sharpe=0.97（接近1.0），但信号灯仍判RED。

**原因** : DSI公式中pass_rate用Sharpe≥1.0计算，EUR区域Alpha普遍较低，导致pass_rate=0，拉低DSI。

#### 问题2：缺少"突破信号"

发现Sharpe=0.97的表达式时，应该给出"加码深挖"的积极信号，而非"结构性改动"。

#### 问题3：DSI权重不合理

当前: DSI = 0.30*S_ttest + 0.25*S_ceiling + 0.25*S_passrate + 0.20*S_consist

问题: S_ceiling权重偏低，高分个体应该更重要；S_passrate权重偏高且有计算缺陷。

建议: DSI = 0.35*S_ttest + 0.35*S_ceiling + 0.15*S_passrate + 0.15*S_consist

#### 问题4：轮次间对比未自动化

Mann-Whitney U检验和Spearman趋势分析需要手动调用，应自动集成到流程中。

### 三、改进方案

#### 改进1：动态调整潜力标准

根据批次实际表现动态调整阈值，而非固定用1.0：

```
if max_sharpe >= 0.9:
    threshold = 0.7  # 发现高分个体时，门槛降低
elif max_sharpe >= 0.5:
    threshold = max_sharpe * 0.7

```

#### 改进2：增加"突破信号"

当max_sharpe >= 0.85时，给出BREAKTHROUGH信号，积极加码。

#### 改进3：简化判断逻辑

用分层判断替代复杂的DSI综合评分：

- 第一层：高分个体优先（max_sharpe >= 0.85 → GREEN）
- 第二层：样本不足保护（n < 10 → YELLOW）
- 第三层：均值显著（mean > 0且显著 → YELLOW）
- 第四层：算子多样性判断（diversity >= 4且无信号 → DEAD）

### 四、创新补充：原帖未涉及的新思路

#### 1. 字段聚类优先级

回测前先对字段聚类，每个聚类用少量表达式"探针测试"，根据结果决定优先深挖方向。

#### 2. 表达式模板库

针对不同字段类型建立最优算子组合：

- 交易计数类: ts_decay_linear + group_zscore
- 持仓价值类: ts_rank + rank
- 集中度类: -rank + ts_mean

#### 3. 阶段止损 vs 全局止损

- 阶段止损: 当前算子组合无效 → 尝试另一种组合
- 全局止损: 多种组合都无效 → 换数据集

#### 4. 回测预算动态分配

GREEN/BREAKTHROUGH: 50%预算 | YELLOW: 20% | RED: 10% | DEAD: 转移预算

## 讨论与评论 (2)

### 评论 #1 (作者: JR57542, 时间: 15天前)

非常详尽的框架验证和改进分析。关于 DSI 权重调整和动态阈值的思路很有参考价值。

我这边在 USA/TOP3000 上使用类似的信号灯系统（我们叫 direction_radar），核心也是四色判断。不过我们的判灯逻辑不太一样：不是计算综合 DSI 分数，而是分层判断——先看 max_sharpe 是否存在突破信号（你说的 BREAKTHROUGH），再看样本深度（有效表达式数量），最后看算子主题多样性。

一个关键差异是：我们对 DEAD 的判定非常保守，需要同时满足「骨架、字段、算子主题都多样化仍疲弱」才判真方向死路。这是因为我们发现很多看似 DEAD 的方向只是骨架单一，换个骨架结构就能复活。具体来说，dead_scope 分为 skeleton_only、field_or_dataset、operator_monotone、family_level 四档，只有 family_level 才建议换经济学逻辑。

关于「信号灯判断过于保守」的问题，我们也遇到过。最终方案是把固定阈值（如 Sharpe≥1.0）改为基于批次内表现的百分位阈值，避免不同 region 的 Sharpe 天花板差异导致误判。
